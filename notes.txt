Why Airflow?
-> Workflow as code: You define pipelines using python (DAG)
-> Scheduling & Dependency Management: It understands task dependencies
-> Centralized Orchestration: One platform control everthing (DGs, APIs, Spark, etc)
-> Monitoring & UI: It has a built in web UI where we can see pipeline status, Debug failures, Trigger runs manually.
Airflow is used because it makes complex, scheduled, multistep workflow reliable, observation and maintanable.

What Airflow?
.Aparche Airflow is an open source tool used to scheduler, manage, and automate workflows especially data pipeline.
.Imagine we have to data pipeline: 
    Get data from API -> Clean the data -> Store in DB -> Generate report
.Instead of running it manually airflow lets us: 
    -> Define all steps
    -> set order (who runs first)
    -> Automate execution

Componenet of Airflow?
-> The metadata DB: kya hua, Kb hua, Kya status hai ka pura record. It's a database (usually postgreSQL or MySQL) where airflow stores everthing about your workflows, not the actual data.
-> The Scheduler: The componenet that decides what to run, when to run, and in what order.
-> The DAG file processor: Wo componenet jo tumahara DAG python files ko read krke unhey airflow ko samajhne layak format me convert krta hai 
-> The Executor: Wo componenet jo tasks ko actually run krta hai 
-> The API Server: It proveds endpoints for task operations and serving the UI
-> The Queue: Tis is list of tasks waiting to be executed.
-> The Trigger: Ti is responsible for managing deferrable tasks - tasks that wait for external events.

Core Concepts: 
-> DAG: Directed \Acylic Graphs is a collection of all the tasks you want to run, organized in a way that reflects their dependencies.
-> Operator: DAG told tat what to do in complete process but operator also inform what to do but only in each steps.
-> Task: Operator ko actual running instance. Operator kya kaam krna hai task jb wo kaam actually run hota hai.
-> Workflow: It is the entire process defined by your DAG, including all tasks and their dependencies.

Single node Archtitecture?
It means that all firflow componenets run on one single machine (one server or VM). In single node architecure, the following componenets all run to the same system:
-> Webserver: UI for monitoring DAGs
-> Scheduleer: Decides when taksk should run
-> Executor: Executes tasks( usually swquental Executor or LocalExecutor)
-> Metadata DG: Often SQLite or locak postgreSQL-> Worker: If lockaexectuor is used, worker run on the same node.

Using Sensor for is_api_available?
A task that waits until some condition becomes true
    from airflow.sdk import dag, task
    from airflow.sdk.bases.sensor import PokeReturnValue
    @task.sensor(poke_interval=30, timeout=300)
        def is_api_available() -> PokeReturnValue:
            response = requests.get("https://dummyjson.com/users/1")

            if response.status_code == 200:
                return PokeReturnValue(
                    is_done=True,
                    xcom_value=response.json()
            )
            return PokeReturnValue(is_done=False, xcom_value=None)

Extract_user?
Third phase of pipeline from where we extract data from response json using pythonOperator
    @task
    def extract_user(fake_user):
        return {
            "id": fake_user["id"],
            "firstname": fake_user["firstName"],
            "lastname": fake_user["lastName"],
            "email": fake_user["email"]
        }

Process_user?
Fourth phase of pipeline
    @task
    def process_user(user_info):
        import csv
        path = "/tmp/user_info.csv"

        with open(path, "w", newline="") as f:
            writer = csv.DictWriter(f, fieldnames=user_info.keys())
            writer.writeheader()
            writer.writerow(user_info)

        return path   # 👈 important (next task ko dena hai)

Store_user?
@task
    def store_user(csv_path):
        hook = PostgresHook(postgres_conn_id="postgres")

        hook.copy_expert(
            sql="""
            COPY users (id, firstname, lastname, email)
            FROM STDIN WITH CSV HEADER
            """,
            filename=csv_path
        )

Creating Multiple assets?
Rather than creating multiple assets seperately we can create multiple asset in single asset
    @asset.multi(
    schedule=[user],
    outlets=["user_location", "user_login"]
    )
    def user_info(user_data):
        result = user_data['results'][0]

        return {
            "user_location": result['location'],
            "user_login": result['login']
        }
